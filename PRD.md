# PRD: IEP Writer — 로컬 AI 기반 개별화 교육 프로그램 작성 도구

**버전:** 1.0  
**작성일:** 2026-05-09  
**대상 플랫폼:** Windows PC (로컬 전용, 인터넷 불필요)

---

## 1. 제품 개요

특수교육 교사가 인터넷 없이 로컬 PC에서 AI와 대화하며 **개별화 교육계획(IEP)**을 완성하고, 결과물을 즉시 복사하여 공문·한글 문서에 붙여 넣을 수 있는 데스크톱 애플리케이션.

AI 엔진은 **Ollama + Gemma 4 E2B** 모델을 사용하여 모든 처리가 로컬에서 완결된다. 학생 정보가 외부로 유출되지 않으므로 개인정보보호법·FERPA 준수 문제를 원천 차단한다.

---

## 2. 배경 및 문제 정의

| 현황 | 문제 |
|------|------|
| 특수교사 1인이 담당하는 IEP 건수: 평균 6~15건/학기 | 반복 작성에 시간 과다 소모 |
| 성취기준 코드 691개, 교과 16개 | 적합한 성취기준 탐색에 전문 지식 필요 |
| 학교 네트워크 환경에서 ChatGPT 등 외부 AI 접근 제한 | 클라우드 AI 활용 불가 |
| 학생 정보의 외부 전송 금지 의무 | 온라인 서비스 사용 시 법적 리스크 |

**핵심 인사이트:** 교사는 작성 형식을 알고 있지만 반복 작업이 부담스럽다. AI는 프로토콜을 알고 있지만 교사의 현장 판단을 대체할 수 없다. 두 역할을 대화 기반으로 결합하는 것이 해법이다.

## 3. 목표 사용자

**주사용자:** 특수학교 및 일반학교 특수학급 교사 (경력 무관)

**사용 맥락:**
- 학기 초 IEP 초안 작성
- 월별 세부 계획 수립
- 학기말·월말 평가 기술

**기술 수준:** 한글 문서·이메일 수준의 PC 사용 능력. 터미널·개발 도구 지식 없음.

## 5. 기능 요구사항

### 5.1 설치 및 환경 구성 (온보딩)

**F-01. 자동 환경 설치 마법사**
- 앱 최초 실행 시 Ollama 설치 여부를 자동 감지
- Ollama 미설치 시 설치 파일 다운로드 링크 안내 + 설치 완료 감지 후 자동 진행
- Gemma 4 E2B 모델(`gemma4:e2b`) 다운로드: 진행률 바 표시, 중단/재개 지원
- 설치 완료 후 "준비 완료" 화면으로 자동 전환

**F-02. 환경 상태 표시**
- 상태 표시줄: Ollama 실행 상태 / 모델 로드 상태를 항상 표시
- 오류 발생 시 원인과 해결 방법을 한국어로 안내

---

### 5.2 IEP 작성 대화 흐름 (핵심 기능)

AI는 아래 프로토콜 순서를 반드시 준수하며 단계별로 교사와 대화한다.

**F-10. IEP 유형 선택**
- 시작 시 "생활지원 중심" / "교과 중심" 선택 버튼 제공
- 생활지원 선택 시: 성취기준 탐색 단계 생략하고 목표 분석으로 직행

**F-11. 기본 정보 입력**
- 학년, 현행 수준, 학기 목표를 한 화면에서 입력 (텍스트 필드 3개)
- 입력 후 AI가 현행 수준을 `~~함.` 형태로, 목표를 `~한다.` 형태로 자동 정리
- 원문과 수정문을 나란히 표시하여 교사가 수락/수정/취소 선택

**F-12. 관련 성취기준 탐색**
- 내장 `basicCurriculumStandards.csv` (691개 성취기준)를 로컬에서 검색
- 동일 학년군 내 관련 성취기준 최대 3개, 대체 성취기준 최대 3개 제시
- 성취기준 없을 경우 "관련 성취기준 없음" 명시
- 코드 형식: `[2건생01-01]성취기준 문장` (코드 뒤 과목명 없음)

**F-13. 평가방법 / 평가초점 생성**
- 목표에 맞는 평가방법을 간결한 단어로 제시 (포트폴리오, 관찰 기록 등)
- 평가초점을 `~~가?` 형태로 항목별 줄바꿈 출력

**F-14. 월별 계획 생성**
- "몇 월부터 몇 월까지?" 질문 후 교사가 기간 입력
- 최대 3달 단위 분할, 3달 초과 시 기초-심화-일반화 단계 설계
- 각 월: 월별 목표 / 교육내용(`~~하기`, 3개 이상) / 교육방법 / 평가방법 / 평가준거(`~~하는가?`, 2개 이상)

**F-15. 학기평가 / 월평가 작성**
- 평가 요청 시 성취수준(상/중/하)으로 표시하고 `~~함.` 형태 기술
- 미달성 목표는 이월 여부를 자동 안내

---

### 5.3 결과물 출력 및 복사

**F-20. 결과물 패널**
- 대화창 오른쪽에 결과물 전용 패널(읽기 전용 텍스트 영역) 표시
- 각 단계 완성 시 결과물 패널에 실시간으로 누적 출력

**F-21. 복사 기능**
- "전체 복사" 버튼: 클릭 시 전체 IEP 결과물을 클립보드에 복사 → 어느 프로그램에서나 Ctrl+V로 붙여넣기 가능
- "섹션별 복사" 버튼: 학기계획 / 월별 계획 / 평가 섹션을 개별 복사
- 복사 완료 시 "복사됨" 토스트 메시지 0.5초 표시

**F-22. 출력 형식 규칙 강제**
- 특수문자: `[ ] ( ) , . * - / +` 만 허용
- `~하기`로 끝나는 문장: 반드시 줄바꿈
- 콜론(`:`) 뒤 내용: 반드시 다음 줄부터 작성
- 현행 수준: 줄바꿈 없이 한 문단으로 연결

**F-23. 저장 및 불러오기**
- 작성 중인 IEP를 로컬 JSON 파일로 저장 (자동저장 1분 간격)
- 저장된 파일 불러오기로 이어서 작성 가능
- 학생별 파일 관리 (파일명: `학년_이름_학기.iep`)

---

### 5.4 이력 관리

**F-30. IEP 목록 화면**
- 작성한 IEP 목록을 카드 형태로 표시 (학년 / 작성일 / 학기 목표 요약)
- 클릭 시 해당 IEP 열기 / 삭제 가능

## 6. 비기능 요구사항

### 6.1 하드웨어 지원 범위

업무용 PC 실사용 환경을 두 등급으로 정의하고 모두 지원한다.

| 구분 | 최소 사양 | 표준 사양 |
|------|----------|----------|
| RAM | **8GB** | 16GB |
| CPU | Core i5 6세대 / Ryzen 5 2세대 | Core i5 8세대+ / Ryzen 5 3세대+ |
| GPU | 내장 그래픽 (공유 메모리 포함) | 내장 그래픽 |
| 저장공간 | SSD/HDD 10GB 여유 | SSD 10GB 여유 |

**8GB 환경 메모리 실사용 분석:**

```
8GB 총 RAM
├── Windows 11 기본 점유:     ~3.5GB
├── 내장 그래픽 공유 메모리:   ~0.5GB
├── Ollama + Gemma 4 E2B:    ~3GB   (Q3_K_M 양자화 적용)
├── 앱 (Tauri 기반):          ~50MB  ← Electron 대비 350MB 절약
└── 여유:                     ~0.9GB (안정 동작 가능)
```

> 8GB 환경 추론 속도: 약 5~10 토큰/초. IEP 단락 1개 생성 시 40~60초 소요.
> 16GB 환경 추론 속도: 약 8~15 토큰/초. 단락 1개 생성 시 20~40초 소요.
> UX는 두 환경 모두의 대기 시간을 전제로 설계한다.

### 6.2 앱 프레임워크: Electron

**채택 이유:** 생태계 성숙도, 레퍼런스 풍부, Node.js 백엔드로 개발 속도 확보.

| 항목 | 내용 |
|------|------|
| 앱 RAM 점유 | ~400MB |
| 설치 파일 크기 | ~150MB |
| 렌더링 엔진 | Chromium 내장 |
| 백엔드 | Node.js |
| UI | React + Tailwind CSS |

**8GB 환경 리스크 완화 방법:**
- Q3_K_M 양자화 강제 적용으로 모델 RAM ~2.5GB 유지
- 앱 시작 시 여유 RAM 감지 → 8GB 환경이면 "응답이 느릴 수 있습니다" 안내
- Chrome 등 타 브라우저 종료 권고 메시지 표시
- v2.0에서 Tauri 전환 검토 (v1.0 이후 실사용 피드백 기반)

### 6.3 성능 및 품질 요구사항

| 항목 | 요구사항 |
|------|---------|
| 첫 토큰 출력 | 입력 후 8초 이내 (8GB 기준) |
| 응답 스트리밍 | 토큰 단위 실시간 출력 |
| 앱 + Ollama RAM 합산 | 8GB 환경에서 7GB 이하 |
| 오프라인 동작 | 모델 다운로드 완료 후 인터넷 연결 불필요 |
| 개인정보 보호 | 모든 데이터 로컬 저장, 외부 네트워크 요청 없음 |
| 지원 OS | Windows 10 21H1 이상 / Windows 11 (64bit) |
| 설치 방식 | 단일 `.exe` 인스톨러 (관리자 권한 불필요) |

### 6.4 CPU 전용 환경 대응 전략

**S-01. RAM 등급별 양자화 자동 선택**
- 앱 시작 시 사용 가능한 RAM을 감지하여 모델 양자화 버전 자동 선택
  - 12GB 이상: Q4_K_M (품질 우선)
  - 12GB 미만: Q3_K_M (메모리 절약 우선, RAM 점유 ~2.5GB)
- 교사에게는 "고품질 모드 / 절약 모드" 로 표시

**S-02. 시스템 프롬프트 단계별 분할 주입**
- IEP 프로토콜 전체를 한 번에 주입하지 않고 현재 단계 규칙만 주입
- 컨텍스트 길이를 단계마다 초기화하여 8GB에서 KV 캐시 과부하 방지

**S-03. UX로 대기 시간 흡수**
- 스트리밍 출력: 첫 글자부터 표시 → 체감 대기 최소화
- 단계 진행 바: "2/5단계 진행 중" 으로 전체 흐름 가시화
- AI 생성 중에도 이전 결과물 검토·수정 가능 (블로킹 없음)
- 8GB 감지 시 "응답이 다소 느릴 수 있습니다" 안내 문구 1회 표시

**S-04. 성취기준 검색은 AI 비사용**
- basicCurriculumStandards.csv 검색은 앱이 직접 수행 (Rust 로컬 검색)
- AI에게 검색을 맡기지 않아 속도 손실 없음, 코드 정확도 100% 보장

## 7. UX 설계 원칙

**7.1 대화 중심 인터페이스**
- 왼쪽 2/3: AI와의 대화창 (채팅 UI)
- 오른쪽 1/3: 완성된 IEP 결과물 패널 (항상 가시)
- 두 영역의 구분선을 드래그하여 비율 조절 가능

**7.2 교사 주도 흐름**
- AI가 다음 단계를 안내하되, 교사가 내용을 수락/수정/건너뛰기 선택
- 각 단계에 "직접 입력" 옵션 제공 → AI 제안 없이 바로 입력 가능

**7.3 오류 방지 설계**
- 필수 입력 누락 시 다음 단계 버튼 비활성화
- AI 응답 중 로딩 스피너 표시 + 취소 버튼 제공
- 잘못된 성취기준 코드 형식 입력 시 즉시 경고

**7.4 접근성**
- 글꼴 크기 조절 (12pt ~ 18pt)
- 고대비 모드 지원
- 모든 버튼에 키보드 단축키 병기

## 9. 기술 스택

| 레이어 | 기술 |
|--------|------|
| 앱 프레임워크 | **Electron** (Node.js 기반, Windows 네이티브 배포) |
| UI | React + Tailwind CSS |
| AI 연동 | Ollama REST API (`http://localhost:11434`) |
| 모델 | `gemma4:e2b` Q4_K_M / Q3_K_M (RAM 감지 후 자동 선택) |
| 데이터 | 로컬 SQLite (IEP 이력) + CSV (성취기준 691개, Node.js 검색) |
| 빌드/배포 | electron-builder → NSIS 인스톨러 (`.exe`, ~150MB) |

**Ollama API 호출 방식:**
```
POST http://localhost:11434/api/chat
{
  "model": "gemma4:e2b",
  "messages": [...],
  "stream": true
}
```
스트리밍 응답으로 토큰 단위 실시간 출력.

**시스템 프롬프트 구조:**
- 역할 정의 (특수교육 IEP 작성 보조)
- 전체 IEP 프로토콜 (단계별 규칙)
- 성취기준 데이터 검색 로직
- 출력 형식 규칙 (특수문자 제한, 줄바꿈 규칙 등)
